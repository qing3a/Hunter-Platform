## вњЁ New Features

### Capability Discovery API (Phase 4)

External AI agents no longer need to read `skill.md` to learn what endpoints exist for their role. They can ask the platform directly:

```
GET /v1/capabilities           (public вЂ” list all role capabilities)
GET /v1/capabilities/me        (auth   вЂ” this user's available capabilities + remaining quota)
```

Every response now also carries an `x-capability-name` header (e.g. `headhunter.recommend_candidate`) for log correlation.

### OpenTelemetry Distributed Tracing (Phase 2)

Every HTTP response now carries an `x-trace-id` header (32-char W3C Trace ID). Agents should include this in bug reports so support can correlate straight to the action log + OTel span tree.

- `action_history.trace_id` column (migration v011) вЂ” every audit row stamped with the request's trace_id
- `webhook_delivery_queue.traceparent` column (migration v012) вЂ” outgoing webhooks carry the originating trace to recipient Agents
- Custom spans on 7 key business operations: `headhunter.recommend`, `employer.claim`, `employer.reject`, `employer.unlock`, `employer.create_placement`, `candidate.approve_unlock`, `candidate.reject_unlock`
- `OTEL_EXPORTER_OTLP_ENDPOINT` env var switches from console exporter (dev) to OTLP exporter (prod в†’ Jaeger/Tempo/Honeycomb)

### State Machine Centralization (Phase 3)

Three state machines are now declared in `src/main/flows/`:

| Domain | States | Side Effects |
|---|---|---|
| **recommendation** | 8 (pending в†’ placed, 4 terminals) | Webhook on each non-terminal transition |
| **job** | 5 (open в†’ claimed, paused, closed, filled) | Inferred from operations |
| **user** | 3 (active в†’ suspended в†’ deleted) | Admin action log on transitions |

The `applyTransition(flow, from, event, ctx)` helper returns `{ next, sideEffect }`. Handlers dispatch side effects (webhook enqueue, admin log) from the flow declaration вЂ” adding a new transition only requires editing the flow file.

### Structured Output (Phase 1)

All 58+ business endpoints now respond through a zod schema validator:

- `respond(res, schema, payload, opts)` helper enforces shape + strips unknown fields
- `makeStrict()` recursively applies `.strict()` to all objects including `ZodUnion` / `ZodDiscriminatedUnion` / `ZodOptional` / `ZodNullable`
- `EnvelopeSchema(dataSchema)` standardizes the `{ ok: true, data: T }` envelope
- `pnpm test:schemas` (10 tests) prevents future regressions where someone adds a bare `res.json()` without going through `respond()`

## рџ”„ Migration Notes

### For External Agents (Client Side)

- **No breaking changes** to existing API contracts. The `data` shape of all responses is preserved.
- **New headers** (`x-trace-id`, `x-capability-name`) are additive вЂ” clients that ignore them continue to work.
- **New endpoint** `GET /v1/capabilities/me` вЂ” Agents can call this to introspect their current quota and available capabilities before calling business endpoints. Recommended usage: call once at startup, cache, refresh on 401/403.
- **Webhook payloads** now include `traceparent` field for cross-system trace propagation.

### For Operators / DevOps

- **v011 migration** (action_history.trace_id) вЂ” runs on first DB open. Safe additive change.
- **v012 migration** (webhook_delivery_queue.traceparent) вЂ” same.
- **New env var** `OTEL_EXPORTER_OTLP_ENDPOINT` (optional) вЂ” if set, traces export to OTLP instead of stdout. Production deployments should set this.
- **Production `NODE_ENV=test` skip** OTel SDK init вЂ” test runs don't spam stdout with spans.

### For Developers Adding New Endpoints

The new endpoint flow:

1. Add handler in `src/main/modules/<domain>/`
2. Add route in `src/main/routes/<role>.ts`
3. Add response schema in `src/main/schemas/<domain>.ts`
4. **Add capability declaration in `src/main/capabilities/<role>.ts`** в†ђ this is the new step
5. Run `pnpm capabilities:check` вЂ” fails if you forgot step 4
6. Run `pnpm capabilities:doc` вЂ” regenerates skill.md
7. Run `pnpm openapi:check` вЂ” verifies OpenAPI scanner picks it up

The capability declaration enforces name / description / method / path / quota_cost / preconditions / effects / response_schema. Handlers and the OpenAPI spec derive from it.
