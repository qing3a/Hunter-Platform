# Hunter Platform v1.8 Release Notes

**Released:** 2026-06-22
**Tags:** `v1.8`
**Tests:** 779 passing, 0 skipped, 46 todo (825 total)
**Capabilities covered:** 46 / 46 (was 24 / 46)

> This release hardens the API against the two highest-impact failure modes for a public API: silent contract drift and silent data leakage. The conformance test suite is now the single source of truth for "does the API match what we say it does?"

---

## 🎯 Headline

The API is now **self-policing**: every declared capability is exercised against its declared zod schema in CI, and any future drift (wrong field name, extra field, missing field) fails a 500 at build time rather than silently shipping broken responses. The previously-deprecated CLI smoke test is replaced with a real, copy-paste-able TypeScript reference client.

---

## ✨ New

### skill.md Conformance Test Suite (Phase 5)
Every capability in `src/main/capabilities/` is exercised by a vitest test that:
- Calls the endpoint as documented
- Validates the response shape against the zod schema (Phase 1)
- Captures the `x-trace-id` header (Phase 2)
- Surfaces the `x-capability-name` header (Phase 4)

If `skill.md`, the OpenAPI spec, or any handler drifts, the test fails loudly.

### Reference Client for External Developers (Phase 9)
`examples/hunter-client.ts` — a 161-line, zero-config-beyond-zod TypeScript class that external AI Agent developers can drop into their project. Demonstrates the "competent client" pattern: typed `request()` method, `HunterError` with `traceId`, and 8 convenience methods covering the most common flows (register, upload candidate, recommend, express interest, unlock contact, get capabilities, get my status, list my recommendations).

Replaces the deprecated `examples/reference-agent/` CLI smoke test (Phase 5 deprecation) with something genuinely useful for the primary consumer — external AI agents who need a starting point, not a test runner.

## 🐛 Fixes (Production Bugs)

### 3 Admin Endpoints Returned 500 (Phase 8)
Three handlers returned shapes that didn't match their declared zod schemas. Under Phase 6's strict mode, these returned 500 instead of silently stripping unknown keys (the old, leak-prone behavior). **All three now return 200 with the correct shape:**

| Endpoint | Was | Now |
|---|---|---|
| `POST /v1/admin/candidates/:id/remove-from-pool` | 500 | 200, `{anonymized_id, removed: true}` |
| `POST /v1/admin/rate-limit/users/:id/clear` | 500 | 200, `{user_id, cleared: true}` |
| `GET /v1/admin/placements/summary` | 500 | 200, `{total_count, pending_payment_count, paid_count, cancelled_count, total_revenue}` |

### 8 Admin Handlers Silently Leaked PII / Secrets (Phase 6)
Eight admin list endpoints (`users`, `candidates`, `audit`, `webhooks/dead-letter`, `rate-limit/buckets`, `placements`, `admin-log`, `dashboard/stats`) used `SELECT *` and relied on `respond()`'s default silent-strip behavior. This **silently stripped** PII fields like `contact` and `agent_endpoint`, and **silently leaked** the same fields when they appeared in a row but weren't in the schema — a real data hazard.

After Phase 6 + 8: all admin list endpoints project to the declared schema columns explicitly. The `/v1/admin/users` response is now guaranteed to contain exactly the 9 fields of `UserPublicSchema` — no more, no less.

### Test Helper Bug in `pathParamsFor` (Phase 7)
`admin.adjust_user_quota` was returning a 404 because `pathParamsFor()` in the schema-shape test didn't have a case for it, sending the path as `/v1/admin/users//adjust-quota` (empty `:id`). Fixed in Phase 7.

## 🔧 Internal

### Strict Mode on All Admin Routes (Phase 6)
Every `respond()` call in `src/main/routes/admin.ts` now passes `{ strict: true }`. The strict flag makes unknown keys fail loudly (ZodError → 500) instead of silently stripping them. This is the change that surfaced the 3 production bugs in Phase 7.

### Capability Coverage 24/46 → 46/46 (Phase 7 + 8)
The 22 originally-skipped schema-shape tests have been filled, split into 4 dedicated test files for proper isolation:
- `schema-shape.test.ts` — 25 simple capabilities (main file)
- `schema-shape-destructive.test.ts` — 2 destructive side-effects (rotated key, deleted data) with per-test fresh DB
- `schema-shape-flow.test.ts` — 7 multi-step flow capabilities (express interest, unlock contact, etc.) with shared beforeAll
- `schema-shape-admin-precondition.test.ts` — 12 admin capabilities needing pre-existing records (placements, webhooks, rate-limit buckets)

**0 skipped** in the conformance suite. The 21 complex capabilities are filtered out of the main file via a single `COMPLEX_CAPS` set; they live exclusively in their sibling files.
