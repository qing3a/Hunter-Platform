## 2. Constraints

- **Backwards compatibility:** `_setup.ts` modification MUST not break the 13 existing conformance test files. Returning extra fields from `freshApp()` is allowed; renaming/removing existing fields is not.
- **Deterministic generator output:** Running `pnpm conformance:gen` twice in a row MUST produce byte-identical output (so CI diffs are reviewable).
- **No production code touched:** This is test-only work.
- **Coverage gate must stay green:** After all changes, `pnpm conformance:check` MUST still report "all 46 capabilities have a scenario test".

### 3.2 Generator unit tests (`tests/unit/scripts/generate-skill-md-scenarios.test.ts`)

5 unit tests:
1. **Output file exists** after running generator
2. **Contains 46 `it.todo(...)` stubs** (one per capability across all 5 sets)
3. **Contains 5 `describe(...)` blocks** for the 5 roles
4. **Every capability name appears** as a string in the output (cross-checked against `getAllCapabilitySets()`)
5. **Idempotent** вҖ” running twice yields byte-identical output

Use `execSync('tsx scripts/generate-skill-md-scenarios.ts')` in `beforeAll`; snapshot-check in `it`s.

### 3.3 schema-shape.test.ts

**Structure:** 5 `describe` blocks (one per role). Inside each, 1 `it()` per capability that has `response_schema`.

**Per-test flow:**
```
1. Find capability via findCapabilityByEndpoint(method, path)
2. If cap.response_schema is undefined вҶ’ it.skip (with reason)
3. Otherwise:
   a. Register appropriate user type (role-aware helper)
   b. Set up minimum prerequisites (e.g., for headhunter.recommend_candidate,
      first POST /v1/headhunter/candidates to get a candidate id)
   c. Call endpoint with supertest via ConformanceClient
   d. Validate r.data against EnvelopeSchema(cap.response_schema)
   e. Assert r.status is in [200, 201, 204]
```

**Helpers (top of file):**
- `setupHeadhunter()` вҶ’ returns `{ hKey, hUserId }`
- `setupEmployer()` вҶ’ returns `{ eKey, eUserId, jobId }`
- `setupCandidate()` вҶ’ returns `{ cKey, cUserId }`
- `setupAuth()` вҶ’ returns noop (auth caps are public-ish)
- `setupAdmin()` вҶ’ returns admin auth header (from `_setup.ts`)

**Skip policy:** Complex multi-step prerequisites (e.g., employer placement needs full recommendation + express-interest + unlock flow) are marked `it.skip` with a `// TODO: requires <flow> prerequisites` comment. Target: at least 30 real tests, вүӨ16 skipped.

### 3.4 failure-modes.test.ts

5 `describe` blocks, one per HTTP status code.

| Status | Trigger | Assertion |
|---|---|---|
| **401** | Call protected endpoint without Authorization header | `r.status === 401`, `r.data.error.code === 'UNAUTHORIZED'` |
| **403** | Call admin endpoint with a non-admin (headhunter) key | `r.status === 401` or `403`, error code `UNAUTHORIZED` or `FORBIDDEN` |
| **409** | Call `POST /v1/candidate/recommendations/:id/approve-unlock` on a `pending` rec (illegal Flow transition) | `r.status === 409`, error code `INVALID_STATE` |
| **429** | DB-direct: set `users.quota_used = users.quota_per_day`, then call a non-zero-cost endpoint | `r.status === 429`, error code `INSUFFICIENT_QUOTA` |
| **500** | `db.close()`, then call any endpoint that hits DB | `r.status === 500` |

Each describe uses its own `freshApp()` to avoid pollution. The 500 test creates `freshApp('failure-500')`; closes `db`; tests one endpoint; test file ends cleanly (no further tests in same describe).

### 3.5 `_setup.ts` modification

Current signature:
```typescript
export async function freshApp(name: string): Promise<{ app: Express; dbPath: string }>
```

New signature (additive, backward-compatible):
```typescript
export async function freshApp(name: string): Promise<{ app: Express; dbPath: string; db: DatabaseSync }>
```

`db` is the open `DatabaseSync` connection. Tests that don't need it just destructure `app` as before.

## 5. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Generator output drifts between runs (non-deterministic) | Pin sort order: roles alphabetical, capabilities by declaration index |
| `_setup.ts` change breaks 13 existing test files | New field is additive; existing destructuring `{ app } = await freshApp(...)` still works |
| schema-shape tests time out due to complex multi-resource setup | Per-test setup helpers; complex flows use `it.skip` with TODO comment |
| 500 test pollutes DB state for other tests in same file | Use separate `freshApp('failure-500')` call in own `describe` block |
| Generated test file produces noisy git diffs | File is auto-generated; only commit when first introduced; subsequent regenerations are mechanical |
| Generator unit tests race against file system | Use `execSync` (synchronous) so beforeAll completes before its run |
